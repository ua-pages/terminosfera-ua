# Terminology Guidelines

Керівництво з укладання термінологічної бази «Терміносфера».

---

## 1. Визначення

### Мова

Визначення пишуться для кожної з трьох підтримуваних мов окремо.

### Структура

- Визначення має бути **одним реченням** (до 20 слів).
- Починається з великої літери, закінчується крапкою.
- Без скорочень, крім загальновживаних (e.g., i.e., etc.).
- Без посилань на інші терміни в тілі визначення (використовувати поле `related`).

### Приклад

```json
{
  "definition": {
    "en": "Storage location for source code, typically used in version control systems like Git.",
    "uk": "Сховище вихідного коду, яке використовується в системах контролю версій, як-от Git.",
    "es": "Ubicación de almacenamiento para código fuente, utilizada en sistemas de control de versiones como Git."
  }
}
```

## 2. Переклади

### Пріоритет джерел

1. Офіційна документація мовою перекладу.
2. Стандарти (ISO, IEEE, ДСТУ).
3. Усталена практика в IT-спільноті.
4. Авторитетні словники (Cambridge, Oxford, RAE, СУМ).

### Принципи

- Переклад має бути **лаконічним** (одне-два слова).
- Якщо термін не перекладається — залишати оригінал (наприклад, `commit` в українській
  та іспанській мовах часто вживається як «коміт» / «commit»).
- Уникати кальок, якщо існує природний відповідник.
- Для української мови дотримуватися норм правопису (2019).

### Варіативність

Якщо термін має кілька варіантів перекладу, основний записується у `translations`,
а альтернативні — у полі `notes` (поки не реалізовано) або в коментарях GitHub Issues.

## 3. Етимологія

### Формат запису

```json
{
  "etymology": {
    "origin": "Latin",
    "root": "repositorium",
    "meaning": "storage place, store"
  }
}
```

- `origin` — мова походження (наприклад, Latin, French, Arabic, Greek, English).
- `root` — кореневе слово в мові оригіналу.
- `meaning` — значення кореня мовою походження (або переклад).

### Де брати етимологію

- Online Etymology Dictionary (etymonline.com).
- Вікісловник (Wiktionary).
- Академічні джерела.

### Коли етимологія не вказана

Якщо походження терміна не встановлено або викликає сумніви — поле `etymology`
залишається порожнім (`{}`). У майбутніх версіях сумнівні випадки проходитимуть
AI-валідацію.

---

## 4. Жаргонізми та рекомендовані відповідники

### Маркування в даних

Для термінів, які мають жаргонні варіанти, використовується структура з позначкою:

```json
{
  "id": "deployment",
  "tags": ["jargon"],
  "translations": {
    "en": "Deployment",
    "uk": "Деплой",
    "es": "Despliegue"
  },
  "recommended": {
    "uk": "Розгортання",
    "es": "Despliegue"
  }
}
```

- Поле `tags` містить `"jargon"`, якщо основний переклад є жаргонізмом.
- Поле `recommended` містить рекомендований нормативний відповідник (якщо відрізняється
  від основного перекладу).
- Якщо `recommended` збігається з `translations`, поле можна опускати.

### Приклади

| Термін | Жаргон | Рекомендовано |
|--------|--------|---------------|
| deployment | деплой | розгортання |
| merge | змержити | злити |
| feature | фіча | можливість / функція |

## 5. Категорії

### Список категорій

Терміни групуються за категоріями. Поточний набір:

- `Git`
- `DevOps`
- `Architecture`
- `Computer Science`
- `Frontend`
- `Backend`
- `Database`
- `Security`
- `Network`
- `Testing`
- `Programming Languages`
- `Design`
- `Protocols`
- `Standards`

Категорія записується в полі `category` англійською мовою.

---

## 6. Пов'язані терміни

### Принципи зв'язування

- У поле `related` записуються `id` термінів, що мають прямий зв'язок.
- Зв'язок має бути симетричним: якщо A пов'язаний із B, то B має містити A у `related`.
- Максимум 5 зв'язків на термін.
- Зв'язки можуть бути:
  - синонімічними (array → list)
  - ієрархічними (repository → commit)
  - асоціативними (cache → deployment)

---

## 7. Формат JSON

### Валідація

Кожен запис терміна проходить перевірку:

1. `id` — лише латиниця, цифри, дефіс. Без пробілів.
2. Усі текстові поля — рядки, без `null`.
3. `related` — масив рядків, можна порожній.
4. `category` — рядок, обов'язкове поле.

### Приклад повного запису

```json
{
  "id": "repository",
  "translations": {
    "en": "Repository",
    "uk": "Репозиторій",
    "es": "Repositorio"
  },
  "definition": {
    "en": "Storage location for source code, typically used in version control systems like Git.",
    "uk": "Сховище вихідного коду, яке використовується в системах контролю версій, як-от Git.",
    "es": "Ubicación de almacenamiento para código fuente, utilizada en sistemas de control de versiones como Git."
  },
  "etymology": {
    "origin": "Latin",
    "root": "repositorium",
    "meaning": "storage place, store"
  },
  "category": "Git",
  "related": ["commit", "deployment"],
  "tags": [],
  "recommended": {}
}
```

## 8. Процес додавання терміна

1. Відкрити Issue із пропозицією терміна.
2. Дочекатися обговорення перекладу та визначення.
3. Додати JSON-файл у `terms/<category>/` і запис у `terms/index.json`.
4. Переконатися, що зв'язки симетричні.
5. Запустити валідацію схеми.
6. Створити Pull Request.
